    Generating the table

    • The table needs to have the class bc-table applied to it
    • By default the tables are assumed to have 10 browsers, if there are more or less a modifier class bc-table-n must be added to the table where n = the number of browser columns. 
    <table class="bc-table bc-table-4"></table>

    Generating the table headings

    • The table headings are wrapped in <thead> tags. 
      <thead></thead>
    • There is an empty <td> at the beginning of both rows.

    • Generating the platform row

      • The <tr> has the class bc-platforms
      • At the moment there are only 2 platforms: desktop and mobile. This is a differentiation that MDN makes, and not one stored in the API. Which browsers belong in which catagory will have to be written into our script.
      • Each <th> in the row must have the class bc-platform-x applied to it where x = the type of platform (desktop or mobile).
      • Each <th> will have a colspan equaling the number of browsers in that catagory
      • Each <th> will have an icon inside indicating the platform type (see icons for more details)
      • Full row example: 
        <tr class="bc-platforms">
  <td></td>
  <th colspan="5" class="bc-platform-desktop">
      <abbr class="only-icon" title="Desktop">
          <span>Desktop</span>

  <th colspan="5" class="bc-platform-mobile">
      <abbr class="only-icon" title="Mobile">
          <span>Mobile</span>
          <i aria-hidden="true" class="ic-mobile"></i>
      </abbr>
  </th>
</tr>

    • Generating the browser row

      • The <tr> has the class bc-browsers
      • Each <th> in the row must have the class bc-browser-x applied to it where x = the browser slug. Browser slugs had not been finalized in the API when I wrote this code and the CSS will need to be updated once the slugs are finalized.
      • When constructing the icons the title and span text will be the browser name. The class for the <i> is ic-x where x = the vendor slug. The vendor slugs are not part of the API.
      • For certian browsers the icon <abbr> needs to include a second <i> with a platform icon, the API also does not contain any information for when to include a platform icon or what they are. (see icons for more details)

    Generating the feature headings

    • Feature headings go in <th>s with the attribute scope="row".
    • The text of the cell is the feature.name.
    • If the name is cannonical it should be wraped in code tags.
    • If the feature has a true value for experimental, standardized, or obsolete the approriate icons should also be inclued, these icons need to be inside a <div class="bc-icons"> (see icons for more details)
    • Full heading example: 
      <th scope="row">
  <code>outterHTML</code>
  <div class="bc-icons">
    <abbr title="Experimental. Expect behavior to change in the future." class="only-icon">
      <span>Experimental</span>
      <i class="ic-experimental" aria-hidden="true"></i>
    </abbr>
  </div>
</th>

    Generating the support cells

    • Each cell should reflect a support object from the API, we should be displaying the most recent support where there was a change.
       

      For example, if version 8, 9, and 10 all have support=yes we should display the support information for version 8.

      If version 8, 9, and 10 are support=yes but version 8 has a footnote, we should display support information for version 9.

      If version 8 and 9 are support=no and version 10 is support=yes we should display the support information for version 10.

      Determining the last version without a change could be complicated, sorry.
    • The most basic support <td> has two classes on it bc-supports-x where x is supports.support and bc-browser-x where x = the browser slug. Browser slugs had not been finalized in the API when I wrote this code and the CSS will need to be updated once the slugs are finalized. 
      <td class="bc-supports-yes bc-browser-firefox"></td>
    • The cell includes one of: Yes, No, Partial, the version number. The version number is preferred, if we have it. There are no special classes or wrappers for this value.

    Generating the support history section and associated pages

    • If there are multiple supports values for a specific feature/browser combo we need to generate a history section within the table cell. These don't display without JavaScript enabled so as a fallback we also need to create a page with the same information we can link to.
    • The two sections are created using nearly identical HTML.
    • The section goes in a <section class="bc-history"> element. If it is going inside a support table add the class hidden and the attriute aria-hidden="true".
       

      The history page

      <section class="bc-history"></section>

      The support table

      <section class="bc-history hidden" aria-hidden="true"></section>

      It should include a support icon. (see icons for more details)

    Generating the Legend

    • The legend immediately follows the compatability table.
    • It is wrapped in: 
      <section class="bc-legend"></section>
    • Legends have a hidden heading: 
      <h3 class="offscreen">Legend</h3>
    • Legends should contain only the symbols that are included in the compatability table they follow (for example, a table with full unqualified support should only have one icon in the table - the icon for full support, like the array compat table below).
    • The legend is a <dl> with each icon in a <dt> and each explanation in a <dd>. The CSS expects only one <dd> after each <dt>. 
      <dt>
  <abbr class="only-icon" title="Requires a vendor prefix.">
    <span>Prefixed</span>
    <i aria-hidden="true" class="ic-prefix"></i>
  </abbr>
</dt>
<dd>
  Requires a <a href="https://developer.mozilla.org/docs/Glossary/Vendor_Prefix">vendor prefix</a>.
</dd>
    • The legend includes slightly different values for the icons than when they appear in a support context. See icons for how to generate the icons.
    • Full example: 
      <section class="bc-legend">
  <h3 class="offscreen">Legend</h3>
  <dt>
    <span class="bc-supports-yes bc-supports">
      <abbr title="Full support" class="bc-level bc-level-yes only-icon">
        <span>Full support.</span>
        &nbsp;
      </abbr>
    </span>
  </dt>
  <dd>Full support.</dd>
  <dt>
    <abbr class="only-icon" title="Requires a vendor prefix.">

    Generating the Footnotes

    • Footnotes have two parts. The icon in the support cell and the corresponding footnotes further down the page.
    • Footnotes should only display information relevant to the supports object that is being displayed in the support cell, footnotes belonging to other, older, browser versions should be handled as part of the history display.
    • For info on generating the icon in the support cell see footnote icons.
    • The footnote section follows the legend section after the end of the compatability table: 
      <section class="bc-footnotes"></section>
    • Each browser which has footnotes gets its own heading which is the browser name + "implementation notes" 
      <h3>Chrome for Desktop implementation notes</h3>
    • The notes collected beneath the heading each have an id defined on them. I've been using the browser slug + feature slug.
       

      If the compatability table includes only one feature these notes can be collected in paragraphs:

      <p class="safari-desktop-datetime">Recognized but no UI</p>

      If the compatability table includes multiple features collect these in a <dl> with the the <dt> listing the applicable feature name and the id on the <dd>.

      <dl>
  <dt>type=datetime</dt>
  <dd id="safari-desktop-datetime">Recognized but no UI</dd>
</dl>
    • Full footnote example: 
      <section class="bc-footnotes">
  <h3>Chrome for Desktop implementation notes</h3>
  <dl>
    <dt>requestpointerlock()</dt>
    <dd id="chrome-desktop-requestpointerlock">
      Allowed in sandboxed <code><iframe></code> if the non-standard value webkit-allow-pointer-lock is set to the sandbox attribute.
    </dd>
  </dl>
  <h3>Notes on Safari for Desktop Compatability</h3>
  <dl>
    <dt>type=datetime</dt>
      <dd id="safari-desktop-datetime">Recognized but no UI</dd>
  </dl>
</section>
